tms_lib documentation

Finite-field digital nets, Sobol matrices, and discrepancy tools.

A compact C++ template library for building and testing (t,m,s) nets and Sobol-style digital sequences over arbitrary supported Galois fields GF(pr).

Quick start API manual Figures References

This page documents the current public API used by main.cpp. In particular, the optimized t-value routine is t_values; the older exhaustive routine is t_values_naive and is mainly useful for debugging.

Quick start

Include tms_lib.h, choose a field through Field<p,r> or a provided alias such as GF2/GF3, build generator matrices, then ask the library for t-values, points, discrepancy, or SVG diagnostics.

#include "tms_lib.h"

typedef GF2 F;

const int dim = 5;
const int matrix_size = 25;
std::vector<Matrix<F>> matrices(dim);
std::vector<MatrixView<F>> views(dim);

for (int d = 0; d < dim; ++d) {
    SobolMatrix<F> sobol(SOBOL_JOE_KUO_GF2, d, matrix_size);
    matrices[d] = sobol.view();      // explicit deep copy from the view
    views[d] = matrices[d].view();
}

std::vector<int> t = t_values(&views[0], dim);
std::vector<double> pts = get_points<F>(&views[0], dim, 512);

double star = star_discrepancy(&pts[0], 512, dim);
double gl2  = generalized_l2_discrepancy(&pts[0], 512, dim);

Most algorithms are template-based and operate directly in the field F. For extension fields, arithmetic is performed through precomputed tables; for prime fields, the scalar type is plain int.

API manual

The entries below document the classes and routines exercised by main.cpp. The emphasis is on finite-field assumptions, computational cost, memory behavior, and safe usage in parallel code.

Defines the scalar type used throughout the library. Prime fields use plain int; extension fields use a compact integer label into precomputed arithmetic tables.

GF2GF3GF4GF5GF8GF16Field<p,r>::T

AssumptionsCoordinates should lie in [0,1); the implementation sanitizes boundary cases for exact star-discrepancy comparisons.
Timegeneralized L2: O(N²d). star: worst-case exponential/combinatorial in dim and N, branch-and-bound prunes heavily in practice. t_value_pointset depends on base-adic occupancy tests.
Memorygeneralized L2 uses O(Nd) plus block temporaries. star discrepancy uses coordinate lists and bit masks, roughly O(dim·N²/word_size) masks in the generic exact solver.

Typical workflows

Test a candidate digital sequence

std::vector<MatrixView<F>> seq(dim);
// fill seq with generator matrices

bool progressive = is_t0_progressive(&seq[0], dim);
std::vector<int> t = t_values(&seq[0], dim);

Use this for matrix-level quality checks. is_t0_progressive is stricter/specialized for t=0; t_values returns the full precision curve.

Evaluate points after randomization

std::vector<double> pts = get_points<F>(&seq[0], dim, npts);

OwenTreeND tree = make_random_owen_tree_nd(dim, base, m, seed);
std::vector<double> scrambled(npts * dim);
apply_owen_permutation_real(&pts[0], &scrambled[0],
                            npts, dim, m, tree);

double gl2 = generalized_l2_discrepancy(&scrambled[0], npts, dim);
double star = star_discrepancy(&scrambled[0], npts, dim);

This is the workflow used to generate the discrepancy plot in the demo.
